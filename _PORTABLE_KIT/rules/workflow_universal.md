# Workflow — универсальный цикл патч→деплой→патчнот→коммит

## Цикл правки

```
правка файла
    │
    ▼
деплой / сборка
    │
    ▼
проверка руками (не доверять "успешному деплою"!)
    │
    ├─ работает ──► патчнот (если визуальный эффект) ──► коммит
    │
    └─ сломалось ──► СТОП ──► откат ──► разобраться с причиной
```

**Важно:** успешный деплой ≠ рабочий результат. Всегда проверять руками после сборки/деплоя.

## Деплой
Команда деплоя специфична для каждого проекта — описывать в проектном `CLAUDE.md`.
Принципы:
- `-NoPrompt` / `--yes` / `-y` — без интерактива
- Перед запуском в телефонном режиме: `powershell -c '$Host.UI.RawUI.WindowSize.Width'` → <80 = молча

## Патчнот → начало `docs/PATCHNOTES.md`

**Писать:** новые фичи, изменения поведения, фиксы видимых пользователю багов.
**НЕ писать:** рефакторинг без изменения поведения, комментарии, структурная уборка.

### Структура патчнота
```
## Патч vX.X.X — YYYY-MM-DD
*Tagline — одна строка суть*

[Симптом / контекст — что было не так или зачем патч]

[Что сделал — что пользователь теперь увидит/почувствует]

[Список с тире — только если несколько отдельных изменений]
```

### Правила тона
- От первого лица: "убрал", "прикрутил", "поймал"
- Без технического жаргона — человеческим языком
- Tagline обязателен (курсив, одна строка)
- Финальная фраза-вывод желательна
- Неформально, мат ок

## Коммит после правки
```
type(scope): краткое описание

Generated with [Claude Code](https://claude.ai/code)
via [Happy](https://happy.engineering)

Co-Authored-By: Claude <noreply@anthropic.com>
Co-Authored-By: Happy <yesreply@happy.engineering>
```
Типы: `feat` / `fix` / `refactor` / `docs` / `style` / `chore`

## Откат
Всегда иметь два уровня отката:
- **prev** — автообновляется при каждом деплое (последняя рабочая версия)
- **backup** — эталон, не меняется без явной команды

Команды отката писать в проектном `CLAUDE.md`.
